While one agent writes code, I am already prompting the next two.

That sentence either sounds like chaos or like the future of software development, depending on where you are in your AI-assisted development journey. Three months ago, I would have said chaos. Today, it is just how I work.

In the previous posts in this series, I covered voice-first prompting, completing the feedback loop, and thinking about prompts as declarative code. Each of those techniques individually made me significantly more productive. But the **compounding effect** of combining all of them -- and then running multiple agents in parallel -- is where things get genuinely surreal.

I built things in the last three months that would have taken **years of focused effort** to build manually. A sync engine framework. A Notion-like document editor. A Figma-style interactive canvas. A full bundle analysis CI pipeline. These are not weekend projects. These are the kinds of systems that would each represent months of work for a senior engineer.

Here is how the multi-agent workflow works, why three is the magic number, and what a truly productive day looks like with this setup.

## The Multi-Directory, Multi-Session Setup

The setup is deceptively simple. I clone my repository into three separate directories:

```
~/projects/
  myapp-1/    # Agent 1's workspace
  myapp-2/    # Agent 2's workspace
  myapp-3/    # Agent 3's workspace
```

Each directory is a full clone of the same repo, on the same branch (or different branches if I am working on unrelated features). I run a separate AI coding agent session in each -- typically Claude Code in separate terminal tabs or windows.

That is the entire infrastructure. Three copies of the repo, three agent sessions, three terminal windows. **No special tooling. No orchestration framework. No shared state between agents.**

The workflow looks like this:

1. **Give Agent 1 a task.** A detailed, voice-dictated prompt with full context, scenarios, and specs. Hit enter.
2. **While Agent 1 works, switch to Agent 2.** Give it a different task. Same level of detail.
3. **While both are working, switch to Agent 3.** Give it a third task.
4. **Agent 1 comes back with questions.** Switch to it, answer the questions, let it continue.
5. **Agent 2 finishes a first draft.** Switch to it, review the output, give feedback.
6. **Agent 3 hits an error.** Switch to it, provide context, let it retry.
7. **Repeat.**

You are essentially playing the role of a **tech lead managing three very fast, very knowledgeable junior developers**. You context-switch between them, but the context-switching cost is low because each task is well-defined (you wrote the spec), and you are reviewing and steering rather than implementing.

## Why Three Is the Sweet Spot

I experimented with different numbers. One agent feels slow -- you spend too much time waiting. Two agents is good but leaves gaps. Four or more agents becomes genuinely chaotic; you cannot keep the context in your head and you start making mistakes in your steering.

**Three agents is the sweet spot** for a few reasons:

**The timing works out.** A typical AI agent task takes 5 to 30 minutes to produce a first draft, depending on complexity. With three agents, by the time you have finished prompting and doing initial review for agents 2 and 3, agent 1 is usually ready for your attention. The rotation keeps you **continuously productive** without dead time.

**Cognitive load stays manageable.** Three concurrent tasks is within most people's working memory capacity. You can keep the high-level state of each task in your head: "Agent 1 is building the notification panel, Agent 2 is refactoring the auth flow, Agent 3 is writing the API integration tests." Four tasks and the mental juggling starts causing errors.

**Git conflicts stay minimal.** With three agents working on the same codebase (or similar feature areas), you occasionally get merge conflicts when combining their work. Three agents produces a manageable number of conflicts. More than three, and conflict resolution starts eating into your productivity gains.

<Callout type="important">
The key insight is that **you are the bottleneck, not the agents.** The agents can work as fast as the model allows. Your job is to keep them supplied with good prompts, timely feedback, and course corrections. Three agents is about the maximum you can keep well-fed without dropping quality.
</Callout>

## Cross-Model Review: Claude Reviews Codex's Work

Here is a technique that sounds ridiculous until you try it: **use one AI model to review another AI model's code.**

I routinely use Claude Code and OpenAI's Codex (or GPT-based coding agents) side by side. Not just for parallel tasks, but for *code review*. The workflow looks like this:

1. Claude writes a feature in directory 1.
2. I copy the relevant files (or point to the diff) and give them to Codex in directory 2.
3. My prompt to Codex: "Claude wrote this code. Review it critically and point out any bugs, edge cases, or architectural issues. Be thorough."
4. Codex produces a review with a list of issues.
5. I feed Codex's review back to Claude: "Codex reviewed your code and found these issues. Address each one."
6. Claude fixes the issues.
7. If needed, I send the fixed code back to Codex for a second pass.
8. Repeat until the issues converge to zero.

Why does this work so well? Two reasons.

**First, different models have different blind spots.** Claude might miss a race condition that Codex catches, while Claude catches a type safety issue that Codex overlooks. Using multiple models gives you **broader coverage**, similar to how multiple human reviewers catch more bugs than a single reviewer.

**Second, the models get competitive.** This sounds anthropomorphic, and maybe it is, but the reviewing model tends to be *thorough* when it knows another model wrote the code. Frame it as "another AI wrote this, find the problems" and the reviewer goes deep. It is like the models want to prove they are better than the other one. Whether this is real or just confirmation bias on my part, the practical result is **extremely thorough code reviews**.

<Callout type="tip">
Frame your cross-model review prompt as: "Another AI wrote this code. Review it critically and find every bug, edge case, and architectural issue." The reviewing model will be noticeably more thorough than if you just asked it to "review this code."
</Callout>

The feedback loop between models converges surprisingly fast. Usually two rounds -- original code, review, fixes, second review -- is enough. By the second review, the issues are down to nitpicks and style preferences, not actual bugs.

## The Essential Safety Net: Codebase Hygiene

Running three agents in parallel against your codebase sounds terrifying if you do not have guardrails. What if one agent introduces a regression? What if two agents make conflicting changes to the same module? What if an agent "refactors" something into an incorrect state?

This is where codebase hygiene goes from "nice to have" to **"non-negotiable."** Here is my setup:

### Linting: Biome

Every agent session runs in a codebase with Biome configured. Biome is fast (it is written in Rust), opinionated, and catches a wide range of issues -- from formatting inconsistencies to actual bugs like unreachable code, unused variables, and suspicious comparisons.

The AI agents can (and do) run `bun x ultracite check` to verify their changes pass linting. If linting fails, they fix the issues before presenting the code for my review. This catches a massive category of low-level mistakes automatically.

### TypeScript: Strict Mode

TypeScript in strict mode is **the single most important safety net** for AI-assisted development. It is not just about catching type errors -- it is about giving the AI an automatic verification mechanism. After writing code, the agent can run `tsc --noEmit` and get immediate feedback about type mismatches, missing properties, incorrect function signatures, and null safety issues.

Without TypeScript, the agent is *guessing*. With TypeScript, it has a **formal contract to verify against**. The difference in output quality is enormous.

### Tests

Tests are the **ultimate verification layer**. When an agent makes changes, it can run the test suite and immediately know whether it broke something. More importantly, when multiple agents are working in parallel and I need to merge their work, the test suite tells me whether the combined result is still correct.

I ask agents to write tests as part of every feature implementation. Not after the feature is done -- *as part of the same task*. "Build the notification panel and write tests for all the scenarios I listed." The AI-generated tests are not always perfect, but they provide a baseline of coverage that makes parallel development safe.

### Git

This almost goes without saying, but: every agent works on a branch (or in a separate directory), and I use git diffs as my primary review tool. I **never** let agents push directly to main. Everything goes through a review workflow -- usually me reading the diff, but sometimes enhanced with cross-model review as described above.

<Callout type="warning">
The combination of linting + types + tests + git is what makes the parallel workflow possible. Without these guardrails, running three agents simultaneously would be reckless. With them, it is just efficient. Do not skip any of these layers.
</Callout>

## The Compounding Effect

Each technique I have covered in this series is valuable on its own. But the real power is in **the combination**. Let me trace through how they compound:

**Voice-first prompting** lets me write detailed, context-rich prompts in 60 seconds instead of 10 minutes. This means I can prompt three agents in the time it used to take me to prompt one.

**Scenario-driven development** gives each agent a clear definition of "done." This means I spend less time steering and more time on the next prompt.

**Complete feedback loops** (browser access, console debugging, dry runs) mean the agents self-correct more often, requiring less intervention from me.

**Declarative specs** constrain the solution space, so the agents produce more architecturally consistent code that merges cleanly.

**Parallel execution** multiplies the output by 3x (or more, since some tasks proceed while I am reviewing others).

**Cross-model review** catches bugs before they reach my review queue, reducing my review time.

Individually, each technique might give a 2-3x improvement. Combined, **the effect is multiplicative**. I am genuinely producing an order of magnitude more working software than I was six months ago. This is not hype. I have the commit history to prove it.

## What a Productive Day Looks Like

Let me walk you through a typical day with this workflow.

**9:00 AM -- Morning planning.** I review my task list and pick three things to work on. Ideally, they are independent enough that the agents will not step on each other's toes. Today: building a user settings page, adding email notification preferences, and writing integration tests for the API layer.

**9:15 AM -- First round of prompts.** I dictate a detailed prompt for Agent 1 (settings page), including the spec document I wrote last night. I switch to Agent 2 and dictate the email preferences prompt. I switch to Agent 3 and dictate the test suite prompt with specific scenarios to cover.

**9:20 AM -- All three agents are working.** I take five minutes to review a colleague's PR, respond to a Slack message, or refill my coffee.

**9:30 AM -- Agent 3 finishes first** (tests are usually fastest). I review the test suite, notice it missed a few edge cases, dictate a follow-up prompt to add them. Agent 3 starts iterating.

**9:35 AM -- Agent 1 asks a clarifying question** about the settings page layout. I answer it and let it continue.

**9:45 AM -- Agent 2 finishes a first draft** of the email preferences feature. I review the diff. The architecture looks good but there is a bug in the unsubscribe flow. I describe the bug and let it fix it.

**10:00 AM -- Agent 1 finishes the settings page.** I send the code to Agent 2 for cross-model review (Agent 2 is done with its fix and now has capacity). Agent 2 finds two issues. I feed them back to Agent 1.

**10:15 AM -- All three tasks are in good shape.** I merge the work from each directory, run the full test suite, fix one merge conflict, and commit.

**10:30 AM -- Second round of tasks.** Three new prompts. The cycle continues.

By the end of the day, I have typically completed **six to nine substantial tasks**. Some of these are full feature implementations. Some are refactors. Some are test suites. Some are infrastructure like CI workflows. The volume is high, but more importantly, the quality is high because every piece has been reviewed, tested, and verified.

A year ago, three tasks per day was a good day. Now, **three tasks by lunch is normal**.

## The Honest Caveats

I want to close this series with some honesty about the limitations and the reality of this workflow.

**AI still produces garbage sometimes.** This has not gone away. Some tasks just do not go well. The agent gets confused, goes in circles, or produces something fundamentally wrong. When this happens, I scrap the attempt, rethink my approach, and try again with a clearer spec. Not every task is a win.

**The learning curve is real.** It took me about three weeks to get comfortable with the parallel workflow. The first week I was constantly confused about which agent was doing what. By the third week, it felt natural. If you try this and it feels chaotic at first, that is normal.

**Not every task benefits from AI assistance.** Deeply creative work, high-level architecture decisions, understanding user needs, defining product strategy -- these are still *fundamentally human activities*. AI accelerates the *implementation* of decisions, not the decisions themselves.

**You need to stay sharp.** There is a risk of atrophying your own coding skills if you lean too heavily on AI. I make a deliberate effort to occasionally work without AI assistance, to stay fluent in the code I am supposedly reviewing. If you cannot understand what the AI is doing, you cannot effectively steer it.

**Context limits are real.** Long agent sessions accumulate context and eventually degrade in quality. I start fresh sessions regularly -- sometimes every 30 to 45 minutes for complex tasks. A fresh session with a good prompt outperforms a stale session with lots of accumulated context almost every time.

<Callout type="tip">
Start fresh agent sessions every 30-45 minutes for complex tasks. A clean session with a well-written prompt will consistently outperform a long-running session with accumulated context drift.
</Callout>

## This Is the New Way

Over the last three months, I built things I genuinely did not think I could build. Not because I lacked the knowledge -- I have been a frontend engineer for over a decade. But because the *time* required to implement complex systems was always the limiting factor. A sync engine is a months-long project. A rich document editor is a months-long project. An interactive canvas is a months-long project.

With the workflow I have described in this series -- voice-first prompting, complete feedback loops, declarative specs, and parallel agents -- each of those became a *weeks-long project*. Not because the AI is magic, but because the AI handles the parts that are time-consuming but not intellectually difficult: the boilerplate, the API glue, the test generation, the pattern implementation, the debugging of straightforward issues.

<Philosophy>
My job has shifted. I spend most of my day thinking about architecture, defining scenarios, writing specs, and reviewing code. Very little of my day is spent actually typing code into an editor. And my output has never been higher.
</Philosophy>

If you have read this entire series, here is my suggestion: **do not try to adopt everything at once.** Start with voice prompting for one week. Then add scenario-driven development. Then set up browser access for your agent. Then try running two agents in parallel. Build the habits incrementally.

The tools will only get better. The models will only get smarter. The developers who learn to steer AI effectively now will have a **compounding advantage** that grows with every model improvement. You do not need to be an AI expert. You need to be a good communicator, a clear thinker, and a thorough reviewer.

Those have always been the most important engineering skills. Now they are also **the most leveraged ones**.
