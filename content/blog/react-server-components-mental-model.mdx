When React Server Components (RSC) first shipped, I struggled with them. Not because the API was complex, but because my mental model was wrong. I kept trying to think about them as "React components that run on the server," when really they represent a fundamental shift in how we compose applications.

## The Two-Tree Model

The key insight is that your React app is no longer a single tree. It's two trees that get stitched together.

Server Components render on the server and send serialized output to the client. Client Components are the React we've always known: they hydrate, handle events, and manage state. The boundary between them is the `'use client'` directive.

## Data Flows Down, Interactivity Stays Local

Here's the pattern that cleared everything up for me. Think of Server Components as the **data layer** and Client Components as the **interaction layer**.

```typescript
// app/posts/page.tsx — Server Component (default)
// This runs on the server. It can access the database directly.
// No useEffect. No loading states. No waterfall.
import { db } from '@/lib/db'
import { PostFeed } from './post-feed'

export default async function PostsPage() {
  const posts = await db.post.findMany({
    where: { status: 'published' },
    include: { author: true, _count: { select: { likes: true } } },
    orderBy: { publishedAt: 'desc' },
  })

  // Pass serialized data to the client boundary
  return <PostFeed initialPosts={posts} />
}
```

```typescript
// app/posts/post-feed.tsx — Client Component
'use client'

import { useState, useTransition } from 'react'
import type { PostWithAuthor } from '@/lib/types'

interface PostFeedProps {
  initialPosts: PostWithAuthor[]
}

export function PostFeed({ initialPosts }: PostFeedProps) {
  const [posts, setPosts] = useState(initialPosts)
  const [filter, setFilter] = useState<string>('all')
  const [isPending, startTransition] = useTransition()

  const filtered = filter === 'all'
    ? posts
    : posts.filter((p) => p.tags.includes(filter))

  return (
    <div>
      <FilterBar
        value={filter}
        onChange={(v) => startTransition(() => setFilter(v))}
      />
      {isPending && <Skeleton count={3} />}
      {filtered.map((post) => (
        <PostCard key={post.id} post={post} />
      ))}
    </div>
  )
}
```

The server fetches the data. The client handles filtering, sorting, and interaction. Clean separation.

## The Serialization Boundary

The `'use client'` directive isn't just a label. It's a **serialization boundary**. Everything that crosses from server to client must be serializable as JSON.

This means you cannot pass:

- Functions or callbacks
- Class instances
- Symbols
- DOM nodes

```typescript
// This will NOT work
async function ServerParent() {
  const handleClick = () => console.log('clicked') // Functions aren't serializable

  return <ClientChild onClick={handleClick} /> // Type error at the boundary
}
```

Instead, think of Client Components as receiving **data** and defining their own **behavior**:

```typescript
// This works perfectly
async function ServerParent() {
  const user = await getUser()

  // Pass data, not behavior
  return <ClientChild userId={user.id} name={user.name} role={user.role} />
}
```

## Composition Patterns

### The Slot Pattern

You can pass Server Components as `children` to Client Components. This is one of the most powerful patterns in RSC:

```typescript
// app/dashboard/layout.tsx — Server Component
import { Sidebar } from './sidebar'
import { DashboardShell } from './dashboard-shell'

export default async function DashboardLayout({
  children,
}: {
  children: React.ReactNode
}) {
  const nav = await getNavigation()

  return (
    <DashboardShell sidebar={<Sidebar items={nav} />}>
      {children}
    </DashboardShell>
  )
}
```

```typescript
// app/dashboard/dashboard-shell.tsx — Client Component
'use client'

import { useState } from 'react'

interface DashboardShellProps {
  sidebar: React.ReactNode // This is a serialized Server Component tree
  children: React.ReactNode
}

export function DashboardShell({ sidebar, children }: DashboardShellProps) {
  const [collapsed, setCollapsed] = useState(false)

  return (
    <div className="flex">
      <aside className={collapsed ? 'w-16' : 'w-64'}>
        <button onClick={() => setCollapsed(!collapsed)} type="button">
          Toggle
        </button>
        {sidebar} {/* Server-rendered content, no hydration cost */}
      </aside>
      <main className="flex-1">{children}</main>
    </div>
  )
}
```

The `sidebar` prop contains server-rendered HTML. The client component can control its visibility without re-rendering the sidebar content. This is zero-cost interactivity wrapping.

### The Data Prefetch Pattern

<Mermaid chart="flowchart TD | A[Server Component] -->|fetch data| B[Database/API] | A -->|pass as props| C[Client Component] | C -->|user interaction| D{Needs fresh data?} | D -->|Yes| E[Server Action] | D -->|No| F[Local State Update] | E -->|revalidate| A | style A fill:#8b5cf6,stroke:#6d28d9,color:#fff | style C fill:#3b82f6,stroke:#2563eb,color:#fff | style E fill:#10b981,stroke:#059669,color:#fff | style F fill:#f59e0b,stroke:#d97706,color:#fff" />

Server Components handle the initial data load. When the user interacts and needs fresh data, Server Actions provide the bridge back to the server without API routes:

```typescript
// app/actions/posts.ts
'use server'

import { revalidatePath } from 'next/cache'
import { db } from '@/lib/db'

export async function likePost(postId: string) {
  const session = await getSession()
  if (!session?.user) throw new Error('Unauthorized')

  await db.like.create({
    data: { postId, userId: session.user.id },
  })

  revalidatePath('/posts')
}

export async function deletePost(postId: string) {
  const session = await getSession()
  if (!session?.user) throw new Error('Unauthorized')

  const post = await db.post.findUnique({ where: { id: postId } })
  if (post?.authorId !== session.user.id) {
    throw new Error('Forbidden')
  }

  await db.post.delete({ where: { id: postId } })
  revalidatePath('/posts')
}
```

## Common Mistakes

### Mistake 1: Making Everything a Client Component

If you add `'use client'` at the top of every file, you've just rebuilt a traditional SPA. Start with Server Components and only push the boundary down to where you need interactivity.

### Mistake 2: Fetching in Client Components by Default

```typescript
// Avoid this pattern when the data is available at request time
'use client'

export function UserProfile({ userId }: { userId: string }) {
  const [user, setUser] = useState(null)

  useEffect(() => {
    fetch(`/api/users/${userId}`).then(r => r.json()).then(setUser)
  }, [userId])

  if (!user) return <Skeleton />
  return <div>{user.name}</div>
}
```

```typescript
// Prefer this: fetch on the server, pass the result
export default async function UserProfile({ userId }: { userId: string }) {
  const user = await db.user.findUnique({ where: { id: userId } })
  if (!user) return notFound()

  return <div>{user.name}</div>
}
```

### Mistake 3: Over-Splitting Components

You don't need a separate Client Component for every `onClick`. Group related interactive elements together to minimize the number of client boundaries.

## The Mental Model, Summarized

Think of your app as a newspaper:

- **Server Components** are the articles, the layout, the images. They're rendered once, read many times, and don't change when you touch them.
- **Client Components** are the interactive crossword puzzle, the video player, the comment section. They respond to user input.

The newspaper analogy works because it captures the key architectural decision: most of your UI is static, data-driven content. Only a small fraction needs interactivity. RSC lets you optimize for that reality.
